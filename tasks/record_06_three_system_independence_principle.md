# 06记录：三系统独立原则

> 本记录正式固定项目层面的"三系统独立原则"，作为后续系统体检 / 重构 / 新 candidate
> 评估的判断标准。
>
> 本轮只写文档，不改代码，不写 DB，不运行 validation，不 commit / push。

---

## 1. 本次记录的背景

continuous_smoothing candidate layer 的 v1 / v2 都已经 abandon。表面看是某个
candidate 单独失败，但回头审视 3R 阶段的全部失败模式，会发现真正的问题不是
"这个 candidate 不好用"，而是更底层的：

> 推演系统、否定系统、置信度系统三者之间的边界没有完全固定。

只要边界没固定，任何新加入的 candidate 都会被默认拉进"看推演结果再否定推演"
的附属层位置，最终都会重复同一类失败：candidate 看似在干否定系统的活，
其实是在被推演结果牵着走。

这次记录的目的，是在进入下一阶段（3R-5 / 3R-6 / 系统体检）之前，先把
三系统的关系、定义、禁止行为写死，让后续所有 prediction / exclusion /
confidence 的开发都站在同一套契约上。

---

## 2. 当前真正的问题

真正的问题不是"推演坏了"，也不是"否定坏了"，而是：

> 三系统没有真正分权。

具体表现：

- 否定系统在多次迭代里逐渐变成了"看推演结果再否定推演"的附属层。
- 否定 candidate 的设计目标，不知不觉从"独立判断最不可能"漂移成"反向校验推演"。
- 置信度的角色一直暧昧：有时被当成"对推演打分"，有时又像"对否定打分"，
  有时又被默认成可以反过来改写推演 / 否定的仲裁者。
- 最终报告层默认拥有"重写任一系统输出"的隐含权力。

只要这些边界不固定，candidate-layer 层级的失败会反复出现，且每次都会被
误诊成"这个 candidate 不行"，而不是真正的 root cause —— 三系统结构本身
没分权。

---

## 3. 推演系统定义

推演系统只回答一个问题：

> **明天最可能发生什么？**

输出：

- 最可能状态
- 次可能状态
- 五状态完整排序
- 推演依据（驱动这个排序的核心信号 / 结构 / 数据）

约束：

- 推演系统不读取否定系统结果。
- 推演系统不被置信度系统改写。
- 推演系统不依赖"否定结果反向喂回"。
- 推演系统的输入只来自原始市场数据 / 历史结构 / 自己的推演规则。

推演系统对自己负责，对"最可能"这件事负责，仅此而已。

---

## 4. 否定系统定义

否定系统只回答一个问题：

> **明天最不可能发生什么？**

输入：

- 原始市场数据
- 自己专属的历史结构、成交额、同行表现、极端形态等数据
- 否定系统自己的规则

输出：

- 最不可能状态
- 否定依据
- 否定强度 / 否定置信度（系统自身评估，不与置信度系统耦合）

约束：

- 否定系统不是"否定推演系统"，它是独立判断"最不可能"的系统。
- 否定系统不读取推演系统结果。
- 否定系统不被推演系统影响。
- 否定 candidate 的合法性，必须基于"它能独立判断最不可能"，
  而不是"它能反向校验推演"。

否定系统对自己负责，对"最不可能"这件事负责，仅此而已。

---

## 5. 置信度系统定义

置信度系统回答的问题：

> **推演系统和否定系统各自这次判断有多可信？**

输入：

- 原始市场数据
- 历史表现
- 推演系统输出（只读）
- 否定系统输出（只读）

输出：

- 推演判断的可信度
- 否定判断的可信度
- 二者一致性 / 冲突程度
- 综合可信度（仅作为展示，不回写）

约束：

- 置信度系统可以观察推演输出和否定输出。
- 置信度系统不能修改推演输出。
- 置信度系统不能修改否定输出。
- 置信度系统不是仲裁者，不能"裁定谁对谁错"，
  只能描述"谁这次更可信"。

置信度系统对"二者各自可信度"负责，不对"最终结论"负责。

---

## 6. 三系统正确关系

正确关系图（文字版）：

```
                市场数据
                   │
     ┌─────────────┼─────────────┐
     ▼             ▼             ▼
  推演系统      否定系统       置信度系统
  （最可能）    （最不可能）   （各自可信度）
     │             │             │
     │             │             │
     └──────┬──────┴──────┬──────┘
            ▼             ▼
            最终报告（仅 aggregate）
```

要点：

- 市场数据**分别**进入推演系统、否定系统、置信度系统。
- 推演系统输出"最可能"。
- 否定系统输出"最不可能"。
- 置信度系统输出二者各自的可信度、一致性、冲突程度、综合可信度。
- 最终报告**只 aggregate**：把三方结果排版、对齐、展示，
  不修改、不改写、不重新裁定。

aggregate 的边界：

- 允许：组合展示、并列呈现、冲突标注、置信度并列、可读化。
- 禁止：替换状态、改写排序、隐藏其中一方结果、用一方覆盖另一方。

---

## 7. 错误关系（未来禁止）

下列模式在未来开发中**严格禁止**，违反即视为破坏三系统独立原则：

1. 推演结果作为否定系统输入
2. 否定系统根据推演结果决定"否定谁"
3. 否定系统改写推演结果
4. 置信度系统改写推演结果
5. 置信度系统改写否定结果
6. final report 改写任一系统结果
7. 在 candidate 评估阶段，用"它能不能反向校验推演"来决定否定 candidate 是否合格
8. 引入跨系统的"反馈回路"，让任意一个系统的输出回流到另一个系统的输入

任何 PR / candidate / 重构如果触发上述模式之一，应被视为违反契约，
默认 reject，除非显式更新本记录。

---

## 8. 与 continuous_smoothing 的关系

continuous_smoothing v1 / v2 已经 abandon as candidate layer。

定性：

- 它**不是 pipeline failure**。
- 它是 candidate-layer 的 **legal fail**：
  作为否定 candidate 评估时，无法独立判断"最不可能"，
  其行为本质上仍依赖"看推演结果再做平滑"。
- 它的失败说明 continuous_smoothing 这一类机制**不适合作为否定 candidate**。

未来重用规则：

- continuous_smoothing 不能再以"否定 candidate"身份回归。
- 如果未来要复用其平滑思路，**只能**作为以下角色独立 launch review：
  - review layer（事后回看）
  - explanation layer（解释推演 / 否定输出）
  - diagnostic layer（诊断哪一个系统这次出了问题）
- 任何"continuous_smoothing 作为否定 candidate"的复活提案，
  在不更新本记录的前提下，默认 reject。

---

## 9. 后续开发原则

下一阶段不要直接进入 3R-5 / 3R-6，也不要继续加新的否定 candidate。

正确顺序：

1. 固定推演系统 contract
   - 输入 / 输出 / 禁止行为 / 失败模式
2. 固定否定系统 contract
   - 输入 / 输出 / 禁止行为 / 否定 candidate 合格标准
3. 固定置信度系统 contract
   - 输入 / 输出 / 只读约束 / 不可改写约束
4. 固定 final report aggregator contract
   - 仅 aggregate / 禁止 mutate / 冲突展示规则
5. 再做系统体检
   - 在三套 contract 已固定的前提下，对照现有代码做合规审计
6. 再决定保留 / 冻结 / 隔离 / 清理
   - 任何不符合 contract 的层都进入冻结或隔离，
     不允许"先改一改就让它过"

约束：

- 在第 1–4 步全部完成之前，不进入 3R-5 / 3R-6。
- 在第 1–4 步全部完成之前，不再新增否定 candidate。
- 在第 1–4 步全部完成之前，不再新增置信度 candidate。
- 推演 candidate 如有新增，需先核对是否符合第 1 步固定的推演 contract。

---

## 10. 最终原则

> 推演系统判断最可能，否定系统判断最不可能，置信度系统判断二者各自可信度。
> 三者互相独立，不互相改写。最终报告只综合展示，不改变任一系统输出。

这一句是后续所有 prediction / exclusion / confidence 开发的判断标准。
任何设计、任何 candidate、任何重构、任何 aggregator 改动，
都必须先通过这一句的检验，再进入实现。
